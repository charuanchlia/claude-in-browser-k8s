Design brainstorm · v1

Claude Code in the browser, one pod per user

A three-tier design that moves the display to a browser and the execution into an isolated Kubernetes pod — so the agentic loop, the tools, and the MCP servers all run server-side, per user, on your local kind cluster.

Decisions locked in this brainstorm Runtime = Claude Agent SDK (TypeScript) in the pod · Lifecycle = ephemeral pods (spawn on connect, delete on disconnect) · Identity = typed username (no auth) · MCP = add-a-server box + live status panel (a mini /mcp) · Transport = WebSocket · Pod creation = gateway calls the k8s API · Packaging = one shared agent-core, two shells (pod for browser + local Mac app) · Credentials = Claude subscription long-lived token (claude setup-token) → k8s Secret → CLAUDE_CODE_OAUTH_TOKEN in the pod.

RESOLVED Auth uses your existing Claude subscription (verified: no ANTHROPIC_API_KEY, OAuth login in Keychain). No API key needed — but "SDK authenticates headless in a pod via the subscription token" is the first spike, since it's the riskiest unknown.

02Why this shape

The split isn't arbitrary — each boundary exists because something can only live on one side of it. Read each row as "what forces this?"

Why a browser for the UI (not a terminal, not a native app)

  • Reach without install. The whole premise is "Claude Code available from a URL." A browser is the one client every user already has — no brew install, no per-OS build. That's the entire reason to move off the terminal.
  • The UI has no reason to be privileged. Rendering chat and streaming tokens needs zero access to a filesystem, shell, or API key. So the client can safely be the least-trusted, most-disposable tier — a thin view.
  • It forces a clean network seam. Because a browser can't run tools, you're pushed into a client/server split with one well-defined channel (the WebSocket). That seam is exactly what lets the backend scale independently.

Why the loop runs in a pod (server-side), never in the browser

  • Secrets. The loop calls the Anthropic API, so it holds a credential. In a browser that key ships to every user and leaks instantly. It must sit somewhere the user can't read — a server process.
  • Capability. "Claude Code" is running commands and editing files. A browser tab has no shell and no filesystem; a pod does. The agent has to run where those exist.
  • MCP. stdio MCP servers are local subprocesses. Only a real OS process (the pod) can spawn and talk to them.
  • Statefulness. An agentic session is long-lived, stateful, and messy (working directory, open MCP connections, conversation memory). That belongs in a durable server process, not a tab that can close mid-loop.

Why one pod per user (not one shared backend)

  • Isolation is the product. Each user gets a shell and filesystem. If two users shared a process, user A's rm -rf or runaway MCP hits user B. A pod is a hard blast-radius boundary — separate filesystem, separate process tree, separate memory.
  • Resource fairness. One user's runaway loop can be capped with pod CPU/memory limits without touching anyone else.
  • Clean lifecycle. "Session over" becomes "delete the pod" — all state and stray processes vanish at once. No cleanup logic to get wrong. This is exactly why ephemeral pods are trivially correct.
  • It's the mechanic you wanted to see. Per-user pods are the literal answer to "how does this scale" — the unit of scaling is the user.

Why Kubernetes at all (not just Docker, not a VM)

  • On-demand scheduling is a solved problem. "Create an isolated environment on demand, place it on a machine with room, tear it down after" is precisely what the k8s API + scheduler do. Rebuilding that by hand over raw Docker is the hard part you'd otherwise write.
  • One API, many machines. On kind it's one node; the same create Pod call spreads across a real cluster later. The code doesn't change as you scale — that's the "B: could be real" property.
  • Batteries you'll want. Secrets, resource limits, namespaces, RBAC, networking — already there, so the gateway stays small.

Why a separate gateway tier (why not let the browser talk to pods directly)

  • Browsers can't reach pods. Pod IPs are internal to the cluster and ephemeral. Something inside the cluster must be the stable, public entry point.
  • Someone must decide "which pod is yours" and create it. That's an authority the client can't have. The gateway owns identity → pod mapping and holds the scoped ServiceAccount that's allowed to create/delete pods.
  • It keeps the k8s API private. Only the gateway talks to the control plane; the client only ever sees one WebSocket. Smaller attack surface, cleaner boundary.
The one-line version UI in the browser because it needs no privilege and must be installable-free · loop in a pod because it holds secrets, needs a shell/filesystem, and is stateful · one pod per user because isolation is the whole product · Kubernetes because on-demand isolated environments is exactly its job · a gateway because someone trusted must route to and mint those pods.

The latency insight you set out to measure The browser↔gateway↔pod hops are in-cluster and local — microseconds to low single-digit ms on kind. The cost you actually feel is (1) pod cold-start, paid once per session because pods are ephemeral, and (2) the pod→Anthropic model call, which is identical to what the terminal Claude Code pays today. Conclusion the demo will prove: moving the UI to a browser adds almost no latency; the interesting number is cold-start, and that's the thing pre-warming (a future experiment) would attack.

Since pods are ephemeral, the MCP config lives in the session's memory — no config files. We'll bake one known-good stdio MCP into the pod image and allow remote-URL MCPs via the box, covering both transports without turning the pod into a package manager.

05Design refinement — one loop, two shells

Now that you want this loop to run both in a pod (browser-driven) and as an app on your Mac, copy Anthropic's actual move: don't build two agents — build one core and wrap it in two thin shells. The core is identical; only the packaging differs.

shared package · agent-core

The loop (write once)

  • Wraps the Claude Agent SDK: the agentic loop, tool execution, MCP client management
  • Exposes a clean interface: startSession(), sendPrompt(), onEvent() (token / tool-call / mcp-status streams), addMcpServer()
  • Knows nothing about WebSockets, HTTP, browsers, or k8s — pure engine
Shell A · runs in the pod

Server shell (browser)

  • Wraps agent-core in a WebSocket server
  • Bridges browser events ↔ core events
  • This is the pod image from §01
Shell B · runs on your Mac

Local shell (app)

  • Wraps the same agent-core
  • Drives it from a local UI (terminal, or a local web/Electron window)
  • Uses your real filesystem — no pod

Appendix — What Claude Code actually does

Verified: the same harness powers both surfaces

Confirmed from Anthropic's public docs (not leaked source): the Claude Agent SDK is the same agent loop that powers Claude Code, exposed as a library — it was renamed from "Claude Code SDK" to "Claude Agent SDK" (~March 2026) for exactly that reason. Terminal Claude Code runs that harness locally on your machine; Claude Code on the web runs each session in an isolated cloud sandbox (filesystem + network isolation, with sensitive credentials kept outside the sandbox). The model always runs on Anthropic's servers.

MCP connection, concretely

On startup Claude Code reads each server from config (a stdio command it spawns, or an http/sse URL it opens), runs the MCP initialize handshake, and discovers that server's tools. Those tools are exposed to the model namespaced as mcp__<server>__<tool>. /mcp is just the surface that shows each server's connection state and lets you reconnect / enable / disable. In our system the Agent SDK does the connect + discovery; we drive it with the three messages in section 03.

Exact Agent SDK option names (e.g. how mcpServers is passed) will be confirmed against the installed SDK when we write the spec — not quoted from memory here.
